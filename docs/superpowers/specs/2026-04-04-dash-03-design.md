# DASH-03: Event Feed — Design Spec

**Date:** 2026-04-04
**Backlog item:** DASH-03
**Status:** Draft

---

## Goal

A scrolling real-time event log showing timestamped detection events as they happen: person detected, product picked up, session ended (with duration), etc.

---

## Approach

A panel on the dashboard page (alongside the unit cards) that receives `session_event` WebSocket messages and prepends them to a capped in-memory list. No new backend work required — the SessionManager already broadcasts these messages (CORE-02).

---

## Placement

The event feed is a right-hand panel on the `/dashboard` page on tablet/desktop:

```
┌──────────────────────────┬───────────────────┐
│  [SensorUnitCard]        │  LIVE EVENTS      │
│  [SensorUnitCard]        │  ─────────────── │
│                          │  14:32:01         │
│                          │  unit-01          │
│                          │  Session ended    │
│                          │  ⏱ 47s  📦 Yes   │
│                          │                   │
│                          │  14:31:12         │
│                          │  unit-01          │
│                          │  Product picked up│
│                          │                   │
│                          │  14:31:00         │
│                          │  unit-01          │
│                          │  Person detected  │
└──────────────────────────┴───────────────────┘
```

On mobile / narrow screens the feed collapses to a bottom drawer or is hidden behind a tab.

---

## Event Display Format

Each entry shows:
- Timestamp (HH:MM:SS, local time)
- Unit name (resolved from unit ID via the store)
- Human-readable event label

| Event type | Display label |
|------------|---------------|
| `session_started` | "Person detected" |
| `session_ended` | "Session ended — ⏱ Xs  📦 Yes/No" |
| `product_picked_up` | "Product picked up" |
| `product_put_down` | "Product put down" |

`session_ended` entries are highlighted with a subtle card border to distinguish them from mid-session events.

---

## State

Feed state lives in the Zustand store (same store as DASH-01):

```typescript
type EventFeedEntry = {
  id: string           // sessionId + event type + ts (for React key)
  unitId: string
  unitName: string
  eventType: string
  ts: string
  dwellSeconds?: number
  productPickedUp?: boolean
}

// In WsStore:
eventFeed: EventFeedEntry[]  // max 200 entries, newest first
```

New events are prepended. When the list exceeds 200 entries, the oldest are dropped. The feed is not persisted — it resets on page reload.

---

## Auto-Scroll Behaviour

- Feed is in a scrollable container.
- If the user has scrolled up to read older entries, do NOT auto-scroll on new events (don't jump their reading position).
- Show a "New events ↓" badge at the bottom when new events arrive while scrolled up. Clicking it scrolls to top (newest).
- If the user is already at the top (newest), auto-scroll is a no-op (prepend is instant).

## Dependencies

No new dependencies.

---

## Acceptance Criteria

- [ ] A `session_started` WebSocket message appears in the feed within 600ms as "Person detected".
- [ ] A `session_ended` message shows duration in seconds and pickup boolean.
- [ ] Feed is capped at 200 entries; oldest are dropped when exceeded.
- [ ] Auto-scroll does not disturb a user who has scrolled up.
- [ ] "New events" badge appears and scrolls to newest on click.
- [ ] Feed resets to empty on page reload (no persistence required).
- [ ] Unit name resolves from unit ID (not just raw ID in display).

---

## Out of Scope

- Filtering the feed by unit or event type
- Persisting the feed across reloads (session history with filter is DATA-01)
- Sound on new events (that is ALERT-01)
