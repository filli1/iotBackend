## Goal

Receive detection events from CORE-01 (DetectionEngine), write completed sessions and their events to the database, and broadcast live updates over WebSocket to the frontend.

---

## Approach

A `SessionManager` class subscribes to `DetectionEvent` callbacks from the engine. It maintains an in-memory map of active sessions (one per unit). On `session_started` it opens a DB record; on `session_ended` it closes it; on pickup events it updates the active session. Every state change is broadcast over WebSocket.

This is the only layer that writes to the database and calls the WebSocket broadcaster â€” the detection engine and ingest route do not.

---

## Responsibilities

| Event received | Action |
|----------------|--------|
| `session_started` | Create `PresenceSession` in DB (status = `active`), store in active map, broadcast `session_event` WS message |
| `session_ended` | Update `PresenceSession` (status = `completed`, `endedAt`, `dwellSeconds`), remove from active map, broadcast `session_event` WS message |
| `product_picked_up` | Update active session `productPickedUp = true`, create `SessionEvent` row, broadcast `session_event` WS message |
| `product_put_down` | Create `SessionEvent` row, broadcast `session_event` WS message |

---

## In-Memory Active Session Map

```typescript
type ActiveSession = {
  sessionId: string
  unitId: string
  startedAt: Date
  productPickedUp: boolean
}

const activeSessions = new Map<string, ActiveSession>() // keyed by unitId
```

Only one active session per unit at a time. A new `session_started` while one is already active (shouldn't happen if the engine is correct) replaces the in-memory entry and logs a warning.

---

## WebSocket Message Types Broadcast

```typescript
// When a session event occurs
{
  type: "session_event"
  unitId: string
  event: "session_started" | "session_ended" | "product_picked_up" | "product_put_down"
  sessionId: string
  dwellSeconds?: number        // only on session_ended
  productPickedUp?: boolean    // only on session_ended
  ts: string                   // ISO timestamp
}
```

The WebSocket broadcaster (DASH-01) also receives raw sensor readings directly from the ingest route â€” the session manager only broadcasts session-level events.

---

## Alert Rule Check

After writing a session event, the session manager checks if an alert should fire:

```typescript
async function checkAlertRule(unitId: string, session: ActiveSession): Promise<void>
```

Reads the unit's `AlertRule` from the database (cached in memory after first load). If the rule is enabled and conditions are met (`dwellSeconds >= dwellThresholdSeconds` and, if `requirePickup`, `productPickedUp === true`), broadcasts an `alert_fired` WebSocket message. The frontend handles display and acknowledgement.

---

## WebSocket Alert Message

```typescript
{
  type: "alert_fired"
  unitId: string
  sessionId: string
  reason: "dwell_threshold" | "pickup" | "dwell_and_pickup"
  ts: string
}
```

## Error Handling

All database operations are wrapped in try/catch. If a DB write fails:
- Log the error
- The active session remains in memory so session tracking continues
- The next successful DB write will attempt to reconcile

Session data is never silently lost â€” the in-memory state persists until the process restarts.

---

## Acceptance Criteria

- [ ] A `session_started` event creates a `PresenceSession` row with status `"active"` and correct `startedAt`.
- [ ] A `session_ended` event updates the row to `status: "completed"`, sets `endedAt` and `dwellSeconds`.
- [ ] A `product_picked_up` event sets `productPickedUp = true` on the active session and creates a `SessionEvent` row.
- [ ] Alert rule is checked after `product_picked_up` and after each dwell update.
- [ ] If alert rule conditions are met, an `alert_fired` WS message is broadcast.
- [ ] Unit tests verify the session lifecycle with a mocked Prisma client and mocked broadcaster.
