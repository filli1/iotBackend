# CORE-01: Detection Engine — Design Spec

**Date:** 2026-04-04
**Backlog item:** CORE-01
**Status:** Draft

---

## Goal

A backend service that receives raw sensor readings from the ingest endpoint and produces presence events. It is the brain of the system — all detection logic lives here.

---

## Approach

A class `DetectionEngine` manages one state machine per sensor unit. State machines are held in memory (one per registered unit). The engine is a pure business-logic layer: it receives readings, updates state, and emits events — it does not write to the database or broadcast to WebSocket directly. Those concerns belong to CORE-02 (SessionManager) and the WebSocket broadcaster respectively.

---

## Per-Unit State Machine

```
IDLE ──► PENDING ──► ACTIVE ──► DEPARTING ──► IDLE
          │ ↑           │            │
          └─┘ (still    └──────────► │ (re-detected
              present               ACTIVE)
              but dwell
              not met)
```

### States

| State | Meaning |
|-------|---------|
| `IDLE` | No person detected |
| `PENDING` | Person detected; waiting for dwell threshold before opening a session |
| `ACTIVE` | Session is open; person is present |
| `DEPARTING` | Person left; waiting for departure timeout before closing session |

### Transitions

| From | To | Condition | Side Effect |
|------|----|-----------|-------------|
| `IDLE` | `PENDING` | `activeSensors >= minSensorAgreement` | Start dwell timer |
| `PENDING` | `ACTIVE` | Dwell timer expires (`dwellMinSeconds`) | Emit `session_started` |
| `PENDING` | `IDLE` | Person leaves before dwell completes | Cancel dwell timer |
| `ACTIVE` | `DEPARTING` | `activeSensors < minSensorAgreement` | Start departure timer |
| `DEPARTING` | `ACTIVE` | `activeSensors >= minSensorAgreement` | Cancel departure timer |
| `DEPARTING` | `IDLE` | Departure timer expires (`departureTimeoutSeconds`) | Emit `session_ended` |

### Presence Detection Logic (per reading)

`activeSensors` = count of ToF sensors where:
- `tof[i].status === "valid"`
- `tof[i].distance_mm <= maxDist[i]` (per-sensor threshold from `TofSensor.maxDist`)
- `tof[i].distance_mm >= minDist[i]` (filters noise from very close objects)

## IMU Event Processing

IMU events are received from `processEvent()` (separate from the sensor reading loop). The engine handles:

| Arduino Event | Detection Engine Action |
|---------------|------------------------|
| `imu_pickup` | If state is `ACTIVE`: emit `product_picked_up` |
| `imu_rotation` | If state is `ACTIVE` and `imuExaminationEnabled`: note for session record |
| `imu_shock` | Log but take no session action |
| `pir_trigger` | Can optionally be used as a presence hint (PIR as wake trigger) |

The engine does NOT re-derive IMU events from raw accel/gyro values — it trusts the hardware-classified events sent by the Arduino. The Arduino's firmware owns IMU classification; the engine only decides what to do with named events.

---

## Emitted Events

The engine calls a registered callback (set by SessionManager) when events occur:

```typescript
type DetectionEvent =
  | { type: 'session_started'; unitId: string; ts: Date }
  | { type: 'session_ended';   unitId: string; ts: Date; dwellSeconds: number }
  | { type: 'product_picked_up'; unitId: string; ts: Date }
  | { type: 'product_put_down';  unitId: string; ts: Date }

type EventHandler = (event: DetectionEvent) => void
```

---

## Interface

```typescript
class DetectionEngine {
  constructor(onEvent: EventHandler) {}

  // Called on startup — loads unit configs from DB to initialise per-unit state
  async loadUnits(): Promise<void>

  // Called by the ingest route for each sensor reading payload
  process(unitId: string, reading: SensorReading): void

  // Called by the ingest route for hardware event payloads
  processEvent(unitId: string, event: HardwareEvent): void

  // Called when a new unit is registered (SETUP-01)
  addUnit(unitId: string, config: UnitConfiguration): void
}
```

## Timers

Dwell and departure timers use `setTimeout`. Each per-unit state machine holds a reference to its active timer and clears it on transition. Node.js single-thread guarantees that timer callbacks and ingest calls do not race — no locks needed.

---

## Configuration Source

On startup, `loadUnits()` reads `UnitConfiguration` rows from the database via the shared Prisma client. In-memory config is used for all real-time decisions; database writes only happen via CORE-02.

If a unit's config is updated via SETUP-02, the engine exposes an `updateConfig(unitId, config)` method that replaces the in-memory state for that unit (without resetting an active session if one is in progress).

---

## Acceptance Criteria

- [ ] Given a stream of readings with `activeSensors >= minSensorAgreement` for `dwellMinSeconds`, a `session_started` event is emitted.
- [ ] Given readings that drop below threshold for `departureTimeoutSeconds`, a `session_ended` event is emitted with correct `dwellSeconds`.
- [ ] If a person leaves before `dwellMinSeconds` elapses, no `session_started` event is emitted.
- [ ] If a person returns during the departure timeout, no `session_ended` is emitted and the session remains active.
- [ ] An `imu_pickup` event while state is `ACTIVE` emits `product_picked_up`.
- [ ] All unit tests pass with Vitest (no real DB or timer required — timers are fake-timed).

---

## Out of Scope

- Writing session data to the database (CORE-02)
- Broadcasting to WebSocket (ws broadcaster)
- Deriving IMU events from raw accel/gyro (the Arduino classifies these)
- PIR-only presence detection (PIR is used as a hint/wake signal, not sole presence indicator)
